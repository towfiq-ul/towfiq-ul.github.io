# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository overview

This is the source for Towfiqul Islam's personal portfolio site (`towfiq-ul.github.io`), deployed via GitHub Pages. The repo contains two generations of the site plus the deployed build output:

- `v2/` — **the active site.** React 19 + TypeScript + Vite. All new work happens here.
- `v1/` — legacy static HTML/CSS/JS version, kept for archival purposes. Do not develop here unless explicitly asked.
- `build/` — the compiled output of `v2` (Vite `outDir` is `../build`). This is what GitHub Pages actually serves from the `master` branch. It's generated — never hand-edit files in it, they'll be overwritten on the next build/deploy.

## Commands (all run from `v2/`)

```bash
npm install          # install deps
npm run start         # vite dev server → http://localhost:3000 (or `npm --prefix v2 run start` from repo root, see run.sh)
npm run build          # tsc (typecheck) && vite build → outputs to ../build
npm run sync            # scrape live site into public/WEBSITE_CONTEXT.md for the AI chat's context (needs .env scrapper vars)
```

There is no separate lint or test script — `npm run build` is the only correctness gate (TypeScript compile via `tsc`, then Vite build). There is no test suite in `v2/`.

## Architecture (`v2/`)

**Navigation is not routing.** Despite `react-router` appearing in some stale prompt docs under `v2/prompts/`, the app does **not** use React Router. `App.tsx` renders one of `Home` / `Contact` / `CV` based on a `currentPage` string in `NavigationContext` (`src/config/navigation-context.tsx`), a simple in-memory state machine, not URL-based routing.

**Content is centralized in data files**, not scattered across components:
- `src/data/portfolio-data.tsx` — personal info, summary, skills, work experience, education, awards, open source. Edit this to update site content.
- `src/data/project-list.tsx` — project entries.

**Styling**: CSS Modules per-component (`*.module.css` colocated with each `.tsx`), plus shared design tokens in `src/styles/tokens/` (colors, spacing, typography, animations, keyframes) pulled into `src/styles/global.css` / `theme.css`. No Tailwind or utility-CSS framework is used. When adding or editing a CSS Module, it must be imported into its component (or the root) to take effect — Vite won't pick it up otherwise.

**AI chat widget** (`src/components/ai/ai-chat.tsx`, the `FloatingChat` component): a floating assistant that answers questions about Towfiqul using a strict grounding contract. On mount it concatenates four context sources — `public/RULESET.md` (the system prompt / persona rules), a parsed CV PDF, `public/WEBSITE_CONTEXT.md` (generated by `npm run sync`), and a markdown knowledge doc — into one system prompt sent with every request. Chat completions are **not** called directly from the browser: the frontend posts to a Cloudflare Worker proxy (`VITE_AI_PROXY_URL`, worker source in `v2/workers/ai-proxy.js`) which holds the real `AI_API_KEY`/`AI_BASE_URL` as Worker secrets and enforces CORS to only the production origin. The assistant's persona and behavioral rules (grounding, tone, contact-intent handling with `[ACTION:EMAIL_ME]` / `[ACTION:WHATSAPP_ME]` tokens) live in `build/RULESET.md` — treat that as the source of truth for chat behavior, not the component code.

**Contact form** uses EmailJS client-side (`VITE_EMAILJS_*` env vars); degrades gracefully (shows an error, rest of the site unaffected) if unconfigured.

**Environment variables** (see `v2/.env.example`): EmailJS keys, AI proxy/model config, and website-scraper credentials (`VITE_SCRAPPER_*`, used only by the `sync` script, not at runtime). Secrets are injected in CI from GitHub Actions secrets (see `.github/workflows/deploy.yml`); the Cloudflare Worker's own secrets (`AI_API_KEY`, `AI_BASE_URL`) are set separately via `wrangler secret put` and never committed.

## Deployment

`.github/workflows/deploy.yml` runs on every push/PR to `master`: installs deps in `v2/`, runs `npm run build` (outputs to `../build`), and deploys `build/` to GitHub Pages via `actions/deploy-pages`. There's no separate staging environment — pushing to `master` ships to production.

The `Makefile` at the repo root is an unrelated versioning helper (git tag bump + push), not part of the app build.

## Working with `v2/prompts/`

This directory holds prompt/instruction docs from a scaffolding tool, but several are stale and describe an architecture the app doesn't actually have (e.g. React Router v7 "framework mode", OpenProps) — verify against actual source (`package.json`, `src/`) before trusting them. The CSS Modules and Lucide-icon-fallback conventions they describe do match current practice.
